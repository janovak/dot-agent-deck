- **Fixed Ctrl+V Splitting Rich Pastes Like Teams Tables**
  Pasting content with embedded tables or other formatting (the classic case: a Teams chat message containing a table, copied into the agent) via **Ctrl+V** previously broke the paste into multiple pieces — the agent would receive the first part, auto-submit on an internal carriage return, then receive the rest into a fresh prompt. Plain text via Ctrl+V was unaffected, and right-click paste already worked. Both paths now behave the same: the Ctrl+V handler reads the full clipboard contents directly (via `arboard`) rather than relying on the per-`Event::Paste` payload that Windows Terminal can split into multiple events for rich content. A 500 ms same-content dedup window then collapses any follow-up paste events that are actually chunks of the same logical Ctrl+V, even if the renderer hangs briefly between them. The decision logic also handles cases where `arboard` succeeds on one chunk and fails on another (it would previously have corrupted the paste — e.g. `abcd` + fallback tail `cd` → `abcdcd`); now an authoritative read suppresses follow-up fallbacks, and a later authoritative read replaces previously-appended fallback bytes when a chunk arrives whose authoritative content starts/ends with the accumulated fallback. The synchronous `arboard` read on the UI thread is now timed; a status message surfaces if it took longer than 50 ms so a stuck Windows clipboard contention case is at least diagnosable rather than presenting as a mystery freeze. On Windows Terminal configurations that swallow **Ctrl+V** entirely and feed the clipboard into the inner pane as a flood of individual key events (no `Event::Paste`), the burst-collector that batches consecutive keys into one bracketed paste now also includes `Tab` as a burstable key and waits 1 ms between events while a burst is in progress — without that wait, the key-Release events that Windows interleaves with key-Press events would break the burst mid-paste and surface as e.g. `C[21 lines]` (the first capital letter as a raw byte, the rest as a separate bracketed paste).
